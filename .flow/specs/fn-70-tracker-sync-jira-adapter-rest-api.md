# fn-70 tracker-sync Jira adapter — Jira REST API + token (Cloud v3 / DC v2, single rung, GitHub-shaped)

## Goal & Context
<!-- scope: business -->

tracker-sync ships **Linear** and **GitHub** adapters; **Jira is detected but deliberately not offered**. Enterprise teams overwhelmingly run Jira and want flow specs mirrored there. This spec adds a Jira adapter implementing tracker-sync's normalized interface so `/flow-next:pilot` backlog mode (fn-68) and every other projection can target Jira with **no special setup beyond a credential the company already issues** — covering Jira **Cloud and self-hosted Data Center/Server**.

**Transport decision (2026-06-28): the Jira REST API + token is the sole supported transport** (Cloud `/rest/api/3` + API-token, DC/Server `/rest/api/2` + PAT) — a single-rung + no-op adapter shaped like the GitHub adapter, NOT a Linear-style MCP ladder. The Atlassian MCP was evaluated and is **not wired**: the *official* Atlassian Remote MCP (GA Feb 2026) is read-mostly — it **cannot transition issue status, update existing fields, or set issue/epic links**, exactly the writes a two-way sync needs; the *community* MCP (`sooperset/mcp-atlassian`) is a REST-wrapper needing its own Docker server + the same PAT, so it adds setup for no capability flow-next's native REST path lacks. (This supersedes the earlier "support MCP too, like Linear" intent, on the evidence.)

## Architecture & Data Models
<!-- scope: technical -->

- **New adapter** behind `references/adapter-interface.md` (nine methods — fn-68 added `listOpenIssues`): the six core (`fetchIssue`/`writeIssue`/`listComments`/`postComment`/`readStatus`/`setStatus`), the **fn-64 relation pair** (`listIssueRelations`/`setIssueRelation`) via Jira native **issue links** ("is blocked by"), and **`listOpenIssues`** — all over the **Jira REST API** (Cloud `/rest/api/3`, DC/Server `/rest/api/2`), mapping the Jira wire shape (incl. ADF) to/from the normalized structs. Reconcile / body-merge / status-sync **who-wins** untouched — with one additive carve-out: the one-way **readiness projection** (status-sync.md § Readiness projection) gains a **Jira branch**, exactly as fn-69 added a GitLab one (see Status-fidelity bullet below).
- **Relations — native links, `linkPresent` always true (fn-69 invariant).** Jira issue links are **native and not license-gated** (no GitLab-style `<!-- flow:deps -->` body-block fallback). So `listIssueRelations` returns each blocked-by edge with **`linkPresent: true`** (the now-**mandatory** `relation` field fn-69 added to `adapter-interface.md` — a native link is itself the visible projection, can never orphan) and **`source: "unknown"`** (Jira links store no flow authorship → the fn-64.1 `depRelations` ledger is the provenance authority, same as Linear/GitHub-native). `setIssueRelation` is **read-before-write deduped** (`listIssueRelations` first; Jira does not reliably no-op a duplicate `POST /issueLink`) and **strictly additive — never deletes** (never-delete-non-ours, R6/R10). A ledgered edge **missing remotely is a human-removal collision** → **defer + receipt, NEVER silently re-create** (the shared steps.md `projectDepRelations` rule, hardened in fn-69 — inherited, not reimplemented).
- **Transport: the Jira REST API + token, single rung + no-op** (GitHub-shaped). The credential/transport is resolved **once at the discovery ceremony** and persisted to config — `env > config`, **no per-run re-probe** (mirrors `cmd_review_backend`). Runtime resolution: **`baseUrl = JIRA_BASE_URL || tracker.perTracker.baseUrl`** (env overrides the persisted config; the persisted `baseUrl` is the default, so config is never inert), credentials from env per the persisted `authScheme` → else **no-op rung** (receipt note, never crash). The only runtime check is a cheap "is the configured credential present" — never a rung re-scan / support-probe. **REST endpoint family by deployment:** Jira **Cloud** = `/rest/api/3` (ADF bodies); **Data Center/Server** = `/rest/api/2`, which may differ in body format (older ADF / wiki markup). **Only Cloud was live-validated (2026-06-28)** — DC/Server stays in scope (the self-hosted requirement) but the adapter branches the API version on the persisted `authScheme`/`apiVersion`, and `references/jira.md` MUST flag the DC/Server endpoint set + body-format deltas as **verify-at-build** (no DC instance was available to validate). **Auth-scheme precedence (rp-review risk):** the **ceremony** picks `authScheme` by deployment — a `*.atlassian.net` `baseUrl` ⇒ `cloud-basic`, else `bearer-pat`; if both `JIRA_API_TOKEN` and `JIRA_PAT` are present **and** the deployment is genuinely ambiguous, **ASK** (never silently guess), then persist. Runtime reads only the persisted `authScheme`, so precedence is decided once at the ceremony, never re-raced per run.
- **Auth (two schemes — detected at ceremony, persisted):** Jira **Cloud** = HTTP basic `email:API_TOKEN` (`JIRA_EMAIL`+`JIRA_API_TOKEN`, an *API token*, not a PAT); Jira **Data Center/Server** = `Authorization: Bearer <PAT>` (`JIRA_PAT`). Label unambiguously so a Cloud user isn't told to make a non-existent "PAT". **Self-hosted TLS:** opt-in `JIRA_SSL_VERIFY=false` for internal-CA/self-signed certs (documented, never silent). Never store creds in flow state — read env each run.
- **Backlog enumeration (`listOpenIssues`) — endpoint by `apiVersion`:** **Cloud v3** = **`POST /rest/api/3/search/jql`** (the legacy `GET /rest/api/3/search` was **removed**, HTTP 410, CHANGE-2046; verified live 2026-06-28) + cursor pagination (below); **DC/Server v2** = `/rest/api/2/search`, whose endpoint + pagination model (likely **offset** `startAt`/`maxResults`, not cursor) are **verify-at-build** — the Cloud 410 evidence does NOT prove DC behavior; DC `listOpenIssues` **IS implemented** via `/rest/api/2/search` (verified at build against a real DC instance), **not deferred** (R1/R9 require all nine methods for DC). Branch on `apiVersion`. Cloud body `{jql, fields, maxResults}`; JQL `project = <projectKey> AND status = "<readyState>"`, where `tracker.readyState` is a **raw Jira status name used directly in the JQL** (e.g. `"Ready for Dev"` — NOT resolved through `statusMap`; it is the tracker-side promoted-lane value, consistent with how Linear/GitHub treat `readyState`), no-op + note when unset. **Cursor pagination** — loop: pass the returned **`nextPageToken` in the next `POST /search/jql` body** until `isLast: true` (NOT offset `total`/`startAt`); bounded with an explicit max + a truncation receipt note. Returns normalized `issue` structs (ADF descriptions translated). Scope JQL to the project (Free-tier rate limits). **JQL safety:** the raw `readyState` status name is **escaped** (quotes/backslashes) before interpolation into `status = "<readyState>"`, and `projectKey` is **validated against the Jira key grammar** before building the query — no JQL injection.
- **ADF body translation** — Jira REST v3 descriptions/comments use Atlassian Document Format. The adapter translates normalized Markdown ↔ ADF at the boundary (Jira-specific; reconcile sees only normalized text). Lossy-but-faithful over a documented subset. **Unknown-node preservation algorithm (required, R3):** never blind-regenerate ADF from markdown (that drops unknown nodes). Instead, on a body write **fetch the current ADF first**, splice the new content **only into the regions that map to the supported markdown subset**, and **carry unknown/unsupported ADF nodes through verbatim** (treat opaque ADF as metadata outside reconcile). (Verified 2026-06-28: description + comment bodies are `{"type":"doc","version":1,"content":[...]}` and round-trip exactly on the supported subset.)
- **Comment `authorAuthority` (fn-68 contract — security-sensitive for the answer valve):** `comment.author.accountType` gives a coarse signal — `app` ⇒ `bot`, `customer` (JSM portal) ⇒ `outsider` — but **`atlassian` does NOT automatically mean `writer`** (an internal Atlassian account isn't necessarily a project writer/approver). For the answer valve, grant **`writer` only with a project-role/group check** (`GET /project/{key}/role` membership) **OR** membership in the configured `tracker.answerAuthors` allowlist; otherwise **`unknown`** (fail closed). **Do NOT introduce a new enum value** — the normalized `authorAuthority` enum is exactly `writer|outsider|bot|unknown`, so a non-authorized Atlassian account maps to `unknown`. The comment author object carries `accountId`/`accountType`/`displayName` but **no inline role** (verified 2026-06-28), so the role lookup is a separate call.
- **Status fidelity — workflow-aware + fn-66 terminal invariant.** Jira has rich per-project workflows. **Status WRITES** map the normalized flow status → a Jira status via `statusMap`, applied through the **transitions API** (`GET /issue/{issueIdOrKey}/transitions` → `POST /issue/{issueIdOrKey}/transitions {transition:{id}}`, preferring the durable `tracker.id`; verified 2026-06-28), never a direct set. **Terminal detection = `status.statusCategory.key`** (`new`|`indeterminate`|`done`), NOT the status *name* (names are project-renamable). Honor the fn-66 invariant: locally-`done` → **In Review** until merge, terminal **Done** gated on `MERGED` PR evidence. **Not every project has the assumed statuses** — the default team-managed workflow is To Do/In Progress/Done with **no "In Review"** (verified 2026-06-28); a missing `statusMap` target or an unreachable transition ⇒ **defer + receipt** (the deterministic default), never assume the In-Review rung exists. **"Nearest" is allowed ONLY via an explicit configured fallback** — never inferred (an inferred nearest risks an illegal/over-eager transition). **`readStatus` (READ normalization — the inverse of the write-map):** reverse-map the Jira status → a normalized status via `statusMap` (match by `id` first, then `name`); else fall back to `statusCategory.key` (`done`→`done`; `new`/`indeterminate`→a safe nonterminal **only when unambiguous**); otherwise return **unmapped + defer receipt** (never guess). Sharp edge: a target transition may be unreachable from the current state → defer + receipt, never a forced/illegal jump.
- **Activation gate (flowctl):** extend `TRACKER_TYPES` to include `"jira"` (deterministic enum edit — the single line that activates `tracker.type: jira`).
- **PR↔issue link (make-pr):** projects to Jira as a **remote link** (or a URL comment) in-adapter — no auto-linkify/`gh`; Smart-Commit keys not relied on.
- **Issue grain:** one flow spec ↔ one Jira issue. The **durable dedupe key is the immutable Jira issue `id`** (stored in `tracker.id`); the display **`key` `PROJ-123` is mutable** (projects can be renamed/moved) and lives in `tracker.identifier`. The link flow accepts a `key`, resolves it to the `id`, and persists the `id`. Back-reference via labeled marker + body anchor.
- **Discovery ceremony — coupled sites (probe / ASK / config-write + readiness):** the probe table (add the `JIRA_BASE_URL`+credential REST signal, flipping today's "surface but don't offer"), the **ASK** step (offer Jira), the **config-write** block (`tracker.type jira` + `perTracker.baseUrl`/`projectKey`/**`authScheme`/`apiVersion`/`statusMap`** — auth scheme + api version detected from the credential/deployment and persisted), and a **Jira readiness branch** (collect `readyState`; validate the status exists in the project when creds are available, else allow skip → no-op backlog lane — mirroring the Linear/GitHub readiness branches). The transport + auth scheme are decided here and persisted — no runtime re-probe.
- **Readiness PROJECTION — a Jira branch in `status-sync.md` (fn-58, additive; the one carve-out to "status-sync untouched").** Distinct from the ceremony (which collects `readyState`) and from the `listOpenIssues` JQL filter (backlog enumeration): the fn-58 readiness projection maps the tracker state onto the local `ready` flag on pull/reconcile. fn-69 added a **GitLab branch** to it (label match + stale-config existence check); fn-70 adds the **Jira branch** — `DESIRED = (issue's status NAME == tracker.readyState)`, a **name match like Linear** (Jira has workflow states, not labels; case-insensitive/trimmed on the status name, not `statusCategory`), plus the **stale-config existence check** (the configured `readyState` status still exists in the project ⇒ genuine not-ready; absent ⇒ warn + noop, never mass-un-ready). One-way, tracker-authoritative, change-only receipts — the same additive shape as fn-69's GitLab branch, NOT a who-wins change.
- **Reference doc:** `references/jira.md`, authored against `references/github.md` (single-rung + transitions-status shape) with the ADF translation + transitions-API status model documented. "enterprise teams", not "portco". Records the **MCP-evaluated-not-wired decision + evidence** so a future spec can revisit without re-researching.

## API Contracts
<!-- scope: technical -->

- **Detection (ceremony, once):** `JIRA_BASE_URL` + (`JIRA_PAT` | `JIRA_EMAIL`+`JIRA_API_TOKEN`) ⇒ REST transport available. Resolution `env > config > ask`; persisted at the ceremony. Surface present AND absent.
- **Auth:** Cloud basic (`email:API_TOKEN`) vs DC/Server Bearer PAT — distinct env vars, labelled in `references/jira.md`.
- **Adapter methods:** exact `adapter-interface.md` signatures; ADF↔Markdown only inside the adapter.
- **Config:** `tracker.type: jira`; `tracker.perTracker` carries `baseUrl`, `projectKey`, **`authScheme` (`cloud-basic` | `bearer-pat` — non-secret; persists the ceremony's deployment decision so runtime never re-infers; credentials still read from env), `apiVersion` (`3` Cloud | `2` DC/Server), optional `sslVerify`,** and `statusMap`. **`statusMap` shape:** a map of **normalized status → an object `{"name": "In Review"}` OR `{"id": "10031"}`** (`id` is preferred over `name` when both are present — names are project-renamable), for the FULL normalized set the adapter emits (`planned`/`in-progress`/`in-review`/`done`/`verified`/…), not just ready/done; e.g. `statusMap: {"in-review": {"name": "In Review"}, "done": {"id": "10031"}}`. **`tracker.readyState` is SEPARATE — a raw Jira status name** (e.g. `readyState: "Ready for Dev"`) used directly in the promoted-lane JQL filter, **NOT** resolved through `statusMap` (consistent with Linear/GitHub treating `readyState` as the raw tracker-side value). A normalized status with **no `statusMap` entry, or whose target is unreachable from the current state, → defer + receipt** (never force); `readyState` unset → `listOpenIssues` no-op.
- **Status writes** go through the **transitions API** (resolve transition id for target status, POST transition).
- **Identity:** durable `tracker.id` = the **immutable Jira issue `id`**; `--identifier PROJ-123` is the display key. `sync set-tracker-id <spec> <id> --identifier PROJ-123 --url <browse-url>` — the link flow accepts a `key`, resolves it to the `id` before persisting. Bare `proj-123` resolves via the widened resolver. (Note: `PROJ-123`/`proj-123` likely already pass the existing `KEY-N` grammar — frame fn-70.1 as **regression tests + error-text/docs**, NOT a risky validator rewrite; preserve GitHub `#N` + reserved `fn`.) **Verify, don't assume (fn-69 scar):** fn-70.1's tests MUST actually exercise `flowctl show PROJ-123` / `work PROJ-123` end-to-end and assert they resolve to the linked spec — fn-69 *assumed* `group/project#12` resolved and it did **not** (`resolve_spec_id_arg`/`parse_any_id` accept only `fn-*` / `KEY-N`). `PROJ-123` IS a `KEY-N`, so it should resolve — but prove it in a test rather than asserting it in prose. **Tracker-first decision (rp-review B2):** because `PROJ-123` IS `KEY-N`, **Jira joins the tracker-first camp like Linear** — BOTH entry flows are supported: **tracker-first** (`spec create --tracker-first --tracker-identifier PROJ-123` → mints canonical `proj-123-slug`) AND **flow-first** (`fn-NN` + `set-tracker-id`). This is **distinct from GitHub `#N` / GitLab `<project>#<iid>`** (flow-first ONLY — not `KEY-N`). **steps.md Phase 2's tracker-first caveat** (fn-69 updated it to "Linear `KEY-N` tracker-first; GitHub/GitLab flow-first") MUST gain a **Jira → tracker-first** line, and `docs/tracker-sync.md`'s grab section likewise (a `.1` deliverable). Mint validation: `spec create --tracker-first --tracker-identifier PROJ-123` must produce a clean `proj-123-slug` (Jira keys are alnum-`-num`, no slugify hazard).

## Edge Cases & Constraints
<!-- scope: technical -->

- **No credential reachable** → no-op rung + actionable receipt ("set `JIRA_BASE_URL` + a token"); spec-first floor still works.
- **ADF round-trips are lossy** — document the supported Markdown subset; preserve unknown ADF nodes on write-back (no silent loss).
- **Transition gating** — a target status may be unreachable from the current status per the workflow; resolve the legal transition, else defer + receipt (never force).
- **Cloud vs Data Center auth differ** — detect by token type at the ceremony; support both.
- **Self-hosted TLS** — opt-in `JIRA_SSL_VERIFY=false` for internal-CA certs; never silent / never default-off.
- **Rate limits / 401 / 403** — bounded retry; auth failure → no-op + actionable receipt; never block. (Free/Standard Cloud tiers have stricter limits — scope JQL to the project.)
- **Cross-platform:** canonical Claude names; `sync-codex.sh` regenerates cleanly.

## Boundaries
<!-- scope: business -->

- **Adapter only** — no reconcile/body-merge/status-sync changes (transport-blind).
- **The Jira REST API + token is the sole transport; no MCP rung** — the official Atlassian MCP can't perform the writes a two-way sync needs (transitions/updates/links), and the community MCP is a redundant PAT-wrapper requiring its own setup. (If an Atlassian MCP happens to be registered, flow-next still uses the REST API — the MCP is neither required nor used.)
- **Not a new sync skill** — plugs into `/flow-next:tracker-sync`.
- **GitHub/Linear/GitLab unaffected** — additive adapter.
- **No Atlassian Connect/Forge app, no MCP dependency** — credentials are a plain API token, nothing to install on the company's Jira.

## Decision Context
<!-- scope: both -->

### Motivation
Enterprise teams run Jira and want flow specs mirrored there. tracker-sync's transport-blind interface means Jira is "just another adapter"; the Jira-specific weight is ADF translation + the transitions-API status model.

### Implementation Tradeoffs
- **Jira REST API + token single-rung, not a Linear-style MCP ladder (decided 2026-06-28, MCP evaluated):** the *official* Atlassian Remote MCP (GA Feb 2026) is read-mostly — Atlassian's own community threads confirm it **cannot transition status, update existing fields, or set issue/epic links** (only basic-issue create, read, comment, JQL). A two-way sync needs exactly those writes, so the MCP can't be the transport. The *community* MCP (`sooperset/mcp-atlassian`) wraps the full REST API but is a third-party Docker server needing the same PAT — redundant with flow-next's native REST path and itself "special setup." So Jira is GitHub-shaped: the Jira REST API + token, single rung. Reverses the earlier "support MCP too, like Linear" intent on the evidence.
- **Transport resolved once at the ceremony, persisted (like review-backend):** no per-run "is the MCP/token good" probing — `env > config`, the no-op rung is the only runtime fallback. Avoids "checking for support all the time."
- **ADF translation in-adapter:** keeps reconcile transport-blind; cost is a documented lossy subset + unknown-node preservation.
- **Transitions API over direct status set:** Jira forbids arbitrary status writes; some target states are unreachable → defer+receipt, never force.
- **Live-API smoke-tested (2026-06-28, dociq.atlassian.net, throwaway project deleted after):** every adapter op validated against real Jira REST v3 — create/read/update issue, ADF `doc` round-trip, transitions, `Blocks` link, remote link, JQL search, colon-labels. Surfaced three corrections now folded in: the `/search`→`/search/jql` removal (410), `statusCategory.key` as the terminal signal (not status name), and that "In Review" isn't universal. Evidence: `~/work/agent-scripts/flow-smoke/out/FINDINGS-jira.md`.
- **Plain API token over Connect/Forge/MCP:** the zero-setup promise — companies already issue API tokens.
